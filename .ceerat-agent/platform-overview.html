Workspace Shape

Major Repositories and Responsibilities

infra

Local stack start/stop, environment wiring, logs, PIDs, database startup, and Makefile targets.

contracts-repo

Protobuf contracts, generated gRPC clients/servers, domain DTOs, mappers, and shared security hooks.

services-repo

Backend services. Currently the mature core is ceerat-user-service.

apps-repo

Admin UI, agent/web UI, customer UI, and the active AI agent service.

ceerat-platform-builder-agent

Codex discovery and consistency tool. Stores durable platform standards in .ceerat-agent.

PostgreSQL

OLTP source of truth for transactional data, owned through backend services.

Typesense

Optional local search index for Career jobs. It is configured from infra/typesense.env and reached only by ceerat-user-service.

Critical Rules

  • Public methods must be rare: login, registration, token validation, and health checks.
  • Customer-owned Career data resolves from JWT user id to customers.user_id; do not trust customer-supplied customer_id.
  • Employment attachment flows verify ownership of both the resume and the reusable employment record.
  • Resume update/delete/download flows fetch by authenticated customer id plus resume id; exports include profile skills and attached employment records.
  • Job search requests from customer surfaces are forced to open jobs; do not trust browser-supplied status for customer search.
  • Market/customer Career metrics are protected service-owned aggregates; apps must not compute broad counts from paginated search or direct database access.
  • Typesense API keys stay server-side in ceerat-user-service; frontend and AI tools must never call Typesense directly.
  • Customer tools and pages must be self-service only.
  • Admin-only operations remain admin-only even if an AI tool would make them convenient.
  • Do not fix permission errors by widening RBAC without checking portal role/session ownership first.

AI Tooling

One Agent Service, Separate Tool Surfaces

Agent/Admin Chat

ceerat-web-ui
  -> /api/agent/chat
  -> ceerat-agent-service /agent/chat
  -> ai.AIThreadService THREAD_PROFILE_AGENT
  -> ToolRunner.Run
  -> agent/admin tools

Tools include customers, services, orders, companies, jobs, career metrics, resume download actions, and application review operations.

Customer Chat

ceerat-customer-ui
  -> /api/chatgpt-client/get-prompt-result
  -> ceerat-agent-service /customer/chat
  -> ai.AIThreadService THREAD_PROFILE_CUSTOMER
  -> ToolRunner.RunCustomer
  -> customer-safe tools

Tools include customer profile, skill profiles, resumes, resume PDF download, career metrics, job search, job cart, and own applications.

search_jobs returns compact job summaries only. It defaults to 10 results, is capped at 20, and excludes full descriptions. Use get_job for one selected job when details are needed.
Persisted AI history stores only sanitized user and final assistant messages. System prompts, tool calls, raw tool results, JWTs, and request metadata must not be stored in thread history.
The AI tool layer is not the security boundary. Tools execute with the user's Ceerat JWT, and backend services still enforce JWT, RBAC, ownership, and repository scoping.

Customer Chat Smoke Test

  1. Log in to ceerat-customer-ui as an active customer.
  2. Open http://localhost:3005/chatgpt-client/ or the customer chat launcher.
  3. Ask “who am I” and confirm get_my_customer_profile succeeds.
  4. Ask “list my skills” and confirm list_my_skill_profiles succeeds.
  5. Confirm an agent/admin account cannot keep a customer portal session.

Search Flow

  1. Customer opens /customer/career/jobs.
  2. Browser calls same-origin GET /api/jobs/search.
  3. ceerat-customer-ui forwards the request to career.JobService/SearchJobs.
  4. ceerat-user-service searches Typesense when enabled and falls back to Postgres when unavailable.
  5. Customers receive open jobs only, with total/page metadata and customer-friendly facets.

Customer Search Surface

Primary Filters

Keyword and location stay visible first for mobile simplicity.

More Filters

Company, work mode, employment type, department, seniority, skills, country, source, and sort live in an expandable area.

Facets

Facet chips show labels and counts such as Databricks (784) and Remote (161). UI copy does not mention Typesense.

Indexed job dimensions include company, location, locations, country, remote, hybrid, onsite, employment type, department, seniority, skills, source, and status. Negative boolean work-mode buckets such as false remote/hybrid/onsite values should be hidden from customer-facing facets.

Other Recent Career Changes

  • Customer Career home is a simple navigation hub with cards for jobs, resumes, skill profiles, job cart, and applications.
  • Career pages follow separate list, create, detail, and edit routes with consistent navigation, breadcrumb, and back actions.
  • Reusable employment records are customer-owned Career records. Resumes attach them through join records instead of duplicating employment history per resume.
  • Career market and customer workflow KPIs come from service-owned aggregate/read-model APIs, not app-side broad count queries.
  • ATS crawlers import universal jobs through ImportATSJobs with an agent token; they do not write directly to Postgres or Typesense.
  • Resume view/download includes profile skills and attached employment records, keeps profile content first, and avoids pipe separators in export text.
  • Greenhouse application flow remains discover, confirm, then submit or manual fallback. UI, crawlers, and AI tools do not submit directly to external ATS providers.

Builder Workflow

How Codex Should Use the Builder

1. Discover

Read context and inventories before planning.

ceerat-builder codex-context
ceerat-builder inventory services
ceerat-builder inventory contracts
ceerat-builder inventory apps

2. Decide

Use standards and evidence to decide ownership, APIs, security, and tests.

ceerat-builder patterns service
ceerat-builder patterns grpc-security
ceerat-builder evidence request "..."

3. Verify

Run targeted builds/tests and drift checks before updating standards.

ceerat-builder verify ...
ceerat-builder check drift
ceerat-builder check apps
Durable builder docs should be updated only after implementation, tests/builds, builder checks, and human validation confirm the pattern is reusable.
Noise rule: treat words like integrate, wire, connect, upgrade, implement, expose, fix, support, and enable as action verbs, not domain names. When inventories already contain a matching RPC or service owner, suppress speculative new proto, database, repository, and service skeleton proposals and focus on the existing owner, callers, RBAC, docs, and verification.
